## 三、完整演示流程与解说稿

## 1. 开场：提出真实学习问题

### 画面操作

先显示作品标题页，随后切换到系统架构图或学习闭环图。

建议展示：

- EduMatrix 智教矩阵；
- 赛题名称；
- “画像—规划—检索—生成—校验—反馈”的闭环；
- Vue 3、FastAPI、Agent Swarm、Hybrid RAG、学情算法、知识图谱等关键词。

### 解说稿

> 大家好，我们展示的作品是 EduMatrix 智教矩阵，参赛赛题是“基于大模型的个性化资源生成与学习多智能体系统开发”。
>
> 传统的智能问答系统通常只根据用户提出的问题生成一段统一答案，但在真实学习场景中，不同学生的专业背景、知识基础、认知风格和学习目标都不相同。
>
> EduMatrix 的核心目标不是简单地回答问题，而是根据学生当前的学习状态，判断他为什么不会、缺少哪些前置知识、适合什么表达方式，然后由多个专业智能体协同生成讲义、思维导图、代码案例、练习题和复习建议。
>
> 系统形成了一个完整的学习闭环：学习者注册和冷启动画像，进入学情分析；系统根据知识图谱和掌握度规划学习路径；Hybrid RAG 检索课程知识和证据；多个 Agent 并行生成学习资源；系统进行证据清洗和跨资源一致性校验；学生通过答题、代码和追问产生反馈；反馈再次更新画像和学习路径。
>
> 因此，EduMatrix 不是一个单轮聊天机器人，而是一个面向专业学习场景的个性化资源生成与多智能体协同学习系统。

>

> 如果注册的是另一个偏好代码实操的学生，系统会在资源排序时优先展示代码案例、运行结果和挑战题。

### 解说稿

> 下面进入学习画像页面。
>
> EduMatrix 的画像不是一个简单的“初学者”或“高级用户”标签，而是一个多维学习状态。
>
> 画像包含知识点掌握度、前置知识缺口、误概念、理解能力、迁移能力、认知负荷、学习动机、交互偏好、学习策略和历史反馈等信息。
>
> 页面中的雷达图展示了不同能力维度，知识点区域展示了学生对具体概念的掌握状态。
>

>
> 后续的学习路径和资源排序都会使用这些画像信息。
>
> 在实现层，系统涉及 BKT、EKF/Kalman、DKT、MIRT/IRT、SM-2 以及基于知识图谱的路径规划。它们分别服务于掌握度更新、时序状态估计、题目能力评估、闪卡复习和学习路径推荐。
>
> 这里展示的是算法参与系统决策的工程闭环，不把当前演示数据描述成大规模真实用户训练效果。

## 6. 智能对话主流程

### 画面操作

进入“智能对话”，输入：

```text
为什么 accuracy 很高，但 recall 很低？请用混淆矩阵、一个具体例子和一段 Python 代码解释。
```

发送后保留：

- 问题提交；
- Agent Timeline；
- 进度事件；
- RAG 证据；
- 资源卡片生成；
- 对齐检查；
- 最终回复。

### 解说稿

> 现在进入系统的核心功能：智能对话。
>
> 我们提出一个看似简单、但包含多个知识点的问题：“为什么 accuracy 很高，但 recall 很低？”
>
> 这个问题涉及分类任务、类别不平衡、混淆矩阵、precision、recall 和代码计算。
>
> 系统收到问题后，不会直接让单个模型生成一段文本，而是先经过画像探针识别学习者的知识状态和表达偏好，再由路径规划师分析目标知识点与前置关系。
>
> 接下来，Hybrid RAG 从课程知识库、文本索引、知识图谱和其他证据来源中检索与问题相关的内容。
>
> 检索结果进入证据清洗模块，对来源、相关性、置信度和冲突进行判断。
>
> 清洗后的证据被分发给多个资源 Agent，分别生成讲义、思维导图、代码案例、练习题和视频或 TTS 脚本。
>
> 最后，系统通过跨模态一致性检查，确认讲义、图示、代码和题目之间的核心知识没有明显冲突，再将资源包通过 SSE 流式返回给前端。

## 7. 展示 Agent Timeline

### 画面操作

展开 Agent Timeline，依次指向：

1. 意图识别；
2. 画像探针；
3. 路径规划；
4. 知识检索；
5. 证据清洗；
6. 资源并行生成；
7. 一致性校验；
8. 反馈保存。

### 解说稿

> 这里是本系统的重要创新展示：Agent Timeline。
>
> 系统当前定义了 9 个职责明确的智能体角色：画像探针、路径规划师、量化评估师、教学路由、理论教授、逻辑画师、极客助教、考官和虚拟导演。
>
> 画像探针负责更新学习者画像、弱点和不会原因；路径规划师根据掌握度、目标和知识图谱规划学习路径；量化评估师评估资源质量和学习反馈；教学路由根据问题意图和学习状态决定处理模式。
>
> 理论教授生成有证据约束的专业讲义；逻辑画师生成知识结构和思维导图；极客助教生成代码案例；考官生成分阶练习题；虚拟导演生成视频或 TTS 脚本及推荐信息。
>
> 这些 Agent 并不是简单地把多个聊天机器人并排放在页面上，而是通过统一的输出契约、事件流和一致性校验共同参与一次学习请求。
>
> 如果某个资源 Agent 出现异常，系统可以把该资源标记为不可用或降级，而不是让整个学习请求完全失败。

### 画面操作

打开讲义或理论资源卡片，展示：

- 问题定义；
- 混淆矩阵；
- accuracy、precision、recall 解释；
- 具体例子；
- 公式；
- 引用或证据来源；
- Markdown 和公式渲染。

### 解说稿

> 首先查看理论讲义。
>
> 系统没有只给出一个公式，而是按照学习者的基础，先解释分类任务和混淆矩阵，再说明 accuracy、precision 和 recall 的区别。
>
> 对当前问题而言，如果负样本数量远大于正样本，即使模型几乎只预测负类，accuracy 也可能很高，但真正的正类召回能力依然很低。
>
> 讲义中同时保留公式、案例和知识来源，使学习者既能理解直觉，也能回到正式定义。
>
> 这里的来源字段很重要。系统不是让模型自由发挥后再把结果包装成正确答案，而是将回答与课程知识库中的证据关联起来，方便复核。

### 解说稿

>

>
> 系统会尽量把错题转化为可执行的学习任务，例如重新讲解、生成相似题、创建闪卡或安排复习。
>
> 这比传统错题本多了一层“为什么错”和“下一步怎么学”。

> 系统不会简单地按照教材目录从第一章开始，而是先检查目标知识点的前置闭包。
>
> 如果当前学生想学习模型评估，但混淆矩阵基础不足，系统会先把混淆矩阵和分类任务补到路径中。
>
> 这体现了 ZPD，也就是最近发展区思想：学习内容既不能低于学生当前水平，也不能直接跳到完全无法理解的难度。
>
> 当学生完成答题或产生新的反馈之后，路径可以重新规划。
