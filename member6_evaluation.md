## 二、 核心任务源码审计与现状评估

成员 6 负责的自适应评测与错题本模块由 3 大核心技术任务以及 2 个视觉交互任务构成。以下是各任务目前的源码级现状与审计结论：

### 1. 任务 1：基于 Fisher 信息增益的计算机自适应测验 (CAT) 算法
*   **任务目标**：编写 MIRT (多维项目反应理论) 能力估计数学函数。每次提交答案后重新积分计算后验概率，选择信息增益最大（把区分度拉满）的下一题发送给前端。
*   **最新源码现状**：
    在 [quiz_api.py](file:///d:/project-edumatrix/edumatrix-main/quiz_api.py#L253-L323) 的 `/generate` 路由中，对于自适应推题的实现逻辑如下：
    1. 若未指定知识点，从画像的 `profile.bkt_states` 中寻找评估误差协方差 `p_err` 最大的概念作为测试目标（此状态更新属于**成员 1** 编写的卡尔曼滤波与 BKT 逻辑）。
    2. 难度难度的确定，完全是通过硬编码规则实现：
       ```python
       if frustration > 0.6 or cognitive_load > 0.7:
           difficulty = 'easy'
       elif mastery < 0.3:
           difficulty = 'easy'
       elif mastery < 0.6:
           difficulty = 'medium'
       elif mastery > 0.8:
           difficulty = 'hard'
       ```
    3. 调用 LLM（考官智能体）根据知识点、难度以及画像生成一道简答题。
*   **评委审计评价**：
    *   **硬伤诊断：MIRT 与 Fisher 信息增益完全为 0 实现。** 
    *   代码中没有定义任何项目反应理论（IRT）或多维项目反应理论（MIRT）的单调逻辑回归函数（如 2PL/3PL 模型），没有计算信息函数（Information Function），更谈不上基于 Fisher 信息增益的最大化筛选。
    *   这完全是粗暴的**规则分支（Rule-based if-else）加 LLM 出题的包装包装**。在国赛答辩中，评委只需问一句：*“请展示你们计算 Fisher 信息量矩阵的雅可比矩阵以及对应的后验概率积分公式。”* 成员 6 的技术水分将瞬间被蒸干。

### 2. 任务 2：结合思维链 (CoT) 的数学/代码推导多步判卷与错题反射 (Reflection)
*   **任务目标**：接口接收学生的分步解答。调用 LLM 按照结构化 JSON 吐出每一行公式的判定，并在前端的推导链上高亮出错步骤，自动记录至错题本，提供反射填补。
*   **最新源码现状**：
    在 [quiz_api.py](file:///d:/project-edumatrix/edumatrix-main/quiz_api.py#L369-L503) 的 `/evaluate` 路由中，调用 LLM 返回一个包含 `accuracy_score`、`misconceptions`、`missing_points` 和 `feedback` 的 JSON 包，然后更新画像并将其写入 `DBWrongQuestion` 数据库。
    前端 [WrongQuestionBook.vue](file:///d:/project-edumatrix/edumatrix-main/frontend/src/views/WrongQuestionBook.vue#L270-L304) 则是通过文本直接输出 `feedback` 和 `accuracy_score`。
*   **评委审计评价**：
    *   **硬伤诊断 1：多步判卷与步骤高亮完全退化。**
    *   系统的判卷是一个整体的文本交互过程，根本**没有实现对数学推导式、代码行逐行 AST 抽象语法树解析的多步评估**，更没有在前端将任何错误的中间步骤高亮显示。它仅仅是把 LLM 吐出的文本塞到卡片中展示，属于最平庸的“大模型包装接口”。
    *   **硬伤诊断 2：致命的远程代码执行 (RCE) 漏洞。**
    *   在 [quiz_api.py](file:///d:/project-edumatrix/edumatrix-main/quiz_api.py#L807-L829) 的 `/similar`（相似题验证）中，为了校验大模型生成的 `python_validator` 校验器是否正确，代码竟然直接调用了 Python 原生的 `exec()`：
      ```python
      # 执行 assert 验证
      local_vars = {}
      exec(python_code, {"__builtins__": __builtins__}, local_vars)
      sandbox_passed = True
      ```
      **这是工业级项目中最不可容忍的安全漏洞。** 在这里，`python_code` 来自大模型生成。一旦发生 Prompt 注入，或者大模型发生幻觉输出了类似 `os.system("rm -rf ...")` 的指令，`exec` 会以 FastAPI 进程的权限在宿主机上无阻碍运行！这不仅会造成系统文件丢失，还可能直接泄漏数据库和 API 密钥。
      更讽刺的是，EduMatrix 明明在 [code_exec_api.py](file:///d:/project-edumatrix/edumatrix-main/code_exec_api.py) 中构建了一套高防御性的隔离沙箱（`SandboxProcessRunner`，支持 Docker 和受限 builtins 过滤），但成员 6 在调用时完全将这一沙箱弃置不用，直接在主进程中大开 `exec` 后门。

### 3. 任务 3：自适应评测模拟器与参数概率校准 (MCMC Simulator)
*   **任务目标**：编写模拟测试脚本，设定一组具有虚拟分布的学生。批量跑测自适应出题，拟合 Item Characteristic Curve (ICC，项目特征曲线)，自动修正 SQLite 题库里的区分度系数。

*   **评委审计评价**：
    *   这是严重的**任务漏建与实质性违约**。该任务在实施方案和分工守则中属于核心理论高地，但当前实现度为 **0%**。

## 三、 国赛擂主级标准对照与硬伤分析（不要留情）

要在国赛答辩中展示出“擂主级”风采，自适应评测与沙箱部分必须做到学术严谨、代码安全、交互惊艳。成员 6 模块有以下 4 处必须无情批判的致命硬伤：

### 🚨 硬伤 1：学术泡沫——以 Kalman 状态强行假冒自适应出题（CAT）
*   **问题剖析**：
    自适应测试的核心是利用信息量最大化（Fisher Information）使测试用例尽快收敛。成员 6 既没有给题库中的题目设计任何难度参数（$\beta$）、区分度参数（$\alpha$）和猜测参数（$\gamma$），也没有实现对能力参数（$\theta$）的最大似然估计（MLE）或贝叶斯最大后验估计（MAP）。仅仅通过读取成员 1 算出的 `p_err` 并用 rule-based `if-else` 分支决定难度标签，再依赖 LLM 现场出题。
*   **无情批判**：
    这在严谨的教育测量专家眼中是不可原谅的“伪自适应”。大模型现场出题的难度具有极强的不稳定性（大模型所谓的 "easy" 题目往往在理科场景下比 "hard" 还要繁琐）。缺乏严格概率矩阵控制的 CAT 算法，在比赛答辩演示时，只要评委要求导出来“学生答题收敛到真实能力的贝叶斯估计曲线”，系统便会瞬间露馅，成为整个团队理论基石崩塌的突破口。

### 🚨 硬伤 2：高危安全硬伤——在主线程无防护直接 `exec` LLM 生成代码
*   **问题剖析**：
    在 `quiz_api.py` 的 `generate_similar_quiz` 中，代码直接将 LLM 生成的 `python_validator` 代码扔进了 `exec()` 中运行，且全局变量传入了危险的 `{"__builtins__": __builtins__}`：
    ```python
    exec(python_code, {"__builtins__": __builtins__}, local_vars)
    ```
*   **无情批判**：
    这是**极其低级的网络安全红线错误**。在生产环境中，这可以直接导致服务器被黑客通过 Prompt 注入直接接管控制权。如果大模型输出的 assert 语句被劫持，不仅会让整个系统面临数据泄露的危险，还会由于未加超时限制和内存限制，导致一旦 LLM 产生了带有死循环的代码，FastAPI 主线程将被彻底锁死。既然同伴已经写好了沙箱 `code_exec_api.py`，却依然使用如此粗暴的方式在主线程里运行大模型生成的代码，这绝非擂主水准。

### 🚨 硬伤 3：理论空置——参数概率校准模拟器为 0 实现
*   **问题剖析**：
    项目需要证明其数据驱动特征（Data-driven calibration）。赛题中专门强调了对题库难度的“自校准能力”。
*   **无情批判**：
    这一项缺失使得系统无法通过历史测验数据实现自我进化，题目难度依旧完全依赖于人工给 LLM 提示词里的主观控制参数。在系统层面上，该模块的自适应推荐无法构成“闭环”，降级成了普通的大模型外挂工具箱，在学术深度和商业潜力上极度受限。

### 🚨 硬伤 4：交互简陋——完全违背全局流光设计系统
*   **问题剖析**：
    前端的交互效果如 `WrongQuestionBook.vue`，完全是十年前的粗糙表格和白板卡片，没有任何 3D 折叠效果，连 Spring physics 的抖动反馈也是空白。
*   **无情批判**：
    在“视觉展示”高度内卷的国赛答辩中，精美的 UI 和极致的用户体验能直接为评委提供最强烈的感官视觉刺激。目前这种毫无质感的课设水准 UI，在第一眼上就会让评委判定为“拼凑出来的半成品”，严重拖累了整个 EduMatrix 系统的精美门面。

## 五、 擂主级突围与产业级落地重构路线图

为彻底扫清上述硬伤，实现“课设 Demo ➔ 国赛擂主 ➔ 产业高可用”的本质性跃迁，必须立刻实施以下重构路线图：

### 1. 实现数学级自适应（MIRT & Fisher 信息增益）重构
*   **重构方案**：
    在 `quiz_api.py` 中引入正统的项目反应理论：
    1.  **题库参数化**：为数据库中的题目引入三个核心参数 $\alpha$（区分度）、$\beta$（难度）和 $\gamma$（猜测率）。
    2.  **能力与信息量估计**：编写数学模型类 `AdaptiveTestEstimator`，当学生答完第 $K$ 题后，利用最大后验概率（MAP）更新学生的当前能力估计值 $\theta$。
    3.  **Fisher 信息量最大化选择**：遍历题库中未考概念的题目，计算每道题的 Fisher 信息量 $I_j(\theta)$：
        $$I_j(\theta) = \frac{P_j'(\theta)^2}{P_j(\theta)(1-P_j(\theta))}$$
        选择当前信息量最大（即对当前能力评估区分度最强）的题目作为下一题进行动态派发，彻底废除粗暴的 hardcode 分支。

### 2. 封堵安全后门：将相似题自校验无缝接入沙箱运行
*   **整改方案**：
    在 [quiz_api.py](file:///d:/project-edumatrix/edumatrix-main/quiz_api.py#L820) 中，**必须彻底废除 `exec()` 调用**。
    应当直接引入已有的沙箱单例 `SANDBOX_RUNNER`：
    ```diff
    - exec(python_code, {"__builtins__": __builtins__}, local_vars)
    - sandbox_passed = True
    + # 路由至进程隔离沙箱运行，限制 CPU 时间及最大内存，防止沙箱逃逸与死锁
    + output, error, exec_time = await SANDBOX_RUNNER.run(python_code)
    + if not error and "AssertionError" not in error:
    +     sandbox_passed = True
    + else:
    +     raise Exception(f"沙箱校验异常: {error}")
    ```
    这样大模型生成的 validator 代码将被拦截在 Docker 或独立的 Python 子进程中运行，一旦有死循环或恶意注入指令，直接会被超时机制和内置安全机制熔断隔离，确保了 FastAPI 服务器的安全。

### 3. 构建 MCMC 参数自校准模拟器
*   **重构方案**：
    在 `scratch/` 下新建 `calibrate_mirt_mcmc.py`。
    1. 模拟 1000 个不同能力层级（从高斯分布中采样得出 $\theta \sim N(0,1)$）的虚拟学生实体。
    2. 批量跑测他们对现有题目的自适应作答表现。
    3. 利用马尔可夫链蒙特卡洛（MCMC，可利用 `PyMC3` 或手动编写吉布斯采样）或 EM 算法拟合 ICC 曲线，逆向估计出题目真实的 $\alpha$, $\beta$ 参数，并以批处理的方式写回 SQLite 中，从而实现题库难度的智能自校准与冷启动校准。
