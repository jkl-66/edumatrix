# EduMatrix 比赛答辩演示脚本

> **现场说明**：主流程无需 Docker。若展示实时 Python 代码，使用 `trusted_local` 仅代表本机受限子进程研究演示，必须同时说明“不具备 Docker 容器隔离”；不要把该模式称为生产级安全沙箱。

## 1. 演示前置条件

演示前必须确认：

- 后端依赖安装完成；
- 前端构建并能加载静态资源；
- 数据库和样例数据已备份；
- 无 Docker 核心模式可正常启动；如演示实时代码执行，再确认 Docker sandbox 正常；
- LLM endpoint/API key 可用，或明确切换 deterministic demo；
- Playwright 浏览器已安装，或不演示 PDF 导出；该依赖与 Docker 独立；
- 三组画像、知识库切片和输入输出样例已固定；
- 不展示真实 API key、密码和真实个人数据。

## 2. 推荐七分钟流程

### 0:00-0:25 项目定位

讲解：

> EduMatrix 面向机器学习导论等垂直技能学习场景，解决同一份课程资源无法适配不同基础学习者、专业知识生成缺少证据约束、学习反馈无法形成路径更新三个问题。系统采用画像探针、路径规划、资源生成和效果评估等 Agent 协同完成学习闭环。

展示：项目首页、总体架构图。

### 0:25-1:10 三组差异化画像

展示三份预先固定的 JSON：

1. 视觉型计算机本科生：已有 Python 基础，偏好图示和例子；
2. 跨专业工程学生：矩阵和梯度前置知识不足，需要分步推导；
3. 项目型学习者：目标是完成不均衡分类项目，偏好代码和挑战题。

讲解：

> 三组画像使用同一学习主题，但学习目标、前置知识、认知风格和资源偏好不同，系统后续路径、解释深度和资源排序应发生变化。

### 1:10-2:20 同题差异化答疑

问题：

```text
为什么 accuracy 很高，但 recall 可能很低？
```

操作：分别切换三组画像并发送同一问题。

观察：

- 画像探针事件；
- 路径规划事件；
- RAG 证据事件；
- 资源 Agent 并行事件；
- 资源顺序和难度变化。

如果使用 deterministic fallback，必须明确说：

> 当前演示使用本地确定性降级路径，展示的是流程和资源结构，不把模板结果当作外部大模型评测结果。

### 2:20-3:35 证据和资源生成

展示：

- 召回的知识点路径；
- 证据来源、chunk 和分数；
- 理论讲义；
- 思维导图；
- 代码实操案例；
- 分阶练习题；
- 视频/TTS 脚本或推荐说明。

讲解：

> 资源生成不是单一模型直接输出，而是由不同职责的资源 Agent 并行生成，再由对齐模块检查概念、公式、代码变量和答案的一致性。

注意：`DebateAugmentedRAG` 已接入当前 Swarm 的 LLM 并由异步 `aclean()` 调用。无外部模型的默认验收仍使用 deterministic 证据清洗，演示时不要把 deterministic 结果宣称为真实三方 LLM 辩论效果。

### 3:35-4:50 代码实操（可选增强）

默认无 Docker 验收时，本段展示代码生成、AST 静态安全检查和沙箱状态页。若演示机已按安装备忘录启用 `trusted_local`，可以继续执行本机可信研究演示；若要展示容器级隔离，则使用 Docker 模式。

先执行安全示例：

```python
from sklearn.metrics import confusion_matrix
y_true = [0, 0, 1, 1]
y_pred = [0, 1, 1, 1]
print(confusion_matrix(y_true, y_pred))
```

再展示恶意样例拦截：

```python
import os
os.system("echo unsafe")
```

讲解：

> 系统先做 AST 检查。默认提交模式不启用代码执行，因此不需要 Docker；可信研究演示模式会在受限本地 Python 子进程中运行，并明确提示“不具备 Docker 容器隔离”；Docker 模式才是在无网络、受限资源的容器中执行。`disabled` 或 Docker 不可用时系统明确拒绝执行，不会隐式切换模式。

### 4:50-5:35 测验和反馈

操作：

1. 选择一题答错；
2. 展示判分、错因和掌握度变化；
3. 进入错题本；
4. 生成相似题；
5. 提交正确答案；
6. 展示复习计划或闪卡到期时间。

讲解：

> 反馈会进入答题记录、错题、画像和复习策略，下一次资源生成可以使用新的学习状态。

### 5:35-6:25 学习路径和画像变化

展示：

- 画像雷达；
- 弱点概念；
- 知识图谱和前置关系；
- 学习路径；
- 资源难度或认知负荷。

### 6:25-6:45 工程和安全说明

主动说明当前版本的边界：

- 认证和 `student_id` 绑定必须完成生产整改；
- 用户知识库必须做 owner 隔离；
- Docker 是代码执行的可选增强，Playwright 只在 PDF 导出时需要；核心学习闭环无需 Docker；
- 三项比赛指标需要独立测试数据证明。

这种表达比宣称不存在风险更可信，也便于评委理解工程成熟度。

### 6:45-7:00 指标与收束

展示指标表：

```text
幻觉率：待完成固定知识库和人工标注后计算
适配准确率：待完成三组画像资源标注后计算
知识点覆盖率：待完成固定知识点集合后计算
```

收束：

> EduMatrix 已完成面向机器学习导论的多智能体个性化教育原型和主要工程链路，下一步工作是完成安全边界、干净环境复现和量化指标验证，使系统从可展示原型进一步成为可审计、可部署的领域教育系统。

## 3. 评委常见问题

### Q1：你们为什么需要多个 Agent？

回答：

> 不同任务的约束不同。画像探针负责理解学习者，规划师负责前置关系和路径，资源 Agent 负责不同媒介，评估和对齐模块负责结果检查。角色拆分使流程可观测、可替换，也能把领域证据约束放在生成之前和之后。

### Q2：如何证明不是简单调用大模型？

回答：

> 系统输入包含结构化画像、知识图谱路径、检索证据和学习反馈，输出包含多个资源角色、引用和对齐信息；测验、错题、代码行为还会更新画像和复习策略。我们会用三组画像同题对比和 Agent 事件日志证明决策差异。

### Q3：幻觉率是否已经小于 5%？

回答：

> 这些指标属于可选效果评测维度，不是当前软件杯 A3 赛题的硬性门槛。源码有证据门限、清洗和对齐机制，但正式结果仍必须在固定知识库和标注集上统计，没有完整数据前我们不会把设计目标写成已达成结果。

### Q4：用户文档如何隔离？

回答：

> 当前实现已在用户文档摄入、检索和删除路径加入 owner_id 元数据与服务端 JWT 学生范围过滤。我们仍会用 A 上传、B 检索的负向测试做最终验收，并把尚未覆盖的路由列入限制。

### Q5：不安装 Docker 能不能验收作品？

回答：

> 可以。提交包默认使用 `EDUMATRIX_SANDBOX_MODE=disabled`，登录、画像、对话、知识检索、测验、反馈和报告等核心学习闭环可以运行。若需要无 Docker 演示代码运行，可在可信演示机设置 `EDUMATRIX_SANDBOX_MODE=trusted_local`，但必须说明它没有容器隔离；若需要更强隔离，再设置 `docker` 并准备 Docker。

### Q6：你们的数据是真实用户数据吗？

回答：

> 当前仓库主要是虚拟种子学生、模拟 Peer 画像和课程题库。它们用于演示系统流程，不会被表述为真实调研或真实学习提升实验。正式指标需要另外收集合规、脱敏的评测数据。

## 4. 演示失败预案

| 故障 | 现场处理 |
|---|---|
| 外部 LLM 失败 | 切换确定性演示并说明 fallback |
| SSE 中断 | 展示预录制 Agent 事件日志和固定输出 |
| Docker 不可用 | 核心闭环继续演示；可信研究演示机可显式使用 `trusted_local`，否则关闭实时代码执行 |
| PDF 导出失败 | 展示预先生成且来源可追溯的报告截图 |
| 前端资源加载失败 | 使用构建产物和本地静态演示 |
| 数据库 locked | 使用只读样例库和单用户演示账号 |
