# 非目标、证据等级与完成定义

版本：`M0-v0.1-draft`

## 1. M0 及比赛阶段非目标

| ID | 非目标 | 原因 | 允许的替代 |
|---|---|---|---|
| NG-01 | 不更换 FastAPI/Vue/SQLite 主框架 | 大规模迁移不直接提高赛题闭环可信度 | 在现有框架内建立稳定契约和迁移层 |
| NG-02 | 不以 Agent 数量作为创新或完成度 | 多角色命名不能证明协同和质量 | 展示职责分离、I/O、运行轨迹和裁决证据 |
| NG-03 | 不把点击、阅读时长或“我会了”当掌握 | 行为弱信号不能代替能力证据 | 使用作答、实操、隐藏测试和延迟复测 |
| NG-04 | 不虚构三项指标达标 | 当前缺少冻结数据集、完整人工标注和正式运行 | 先冻结公式和案例，结果按真实运行发布 |
| NG-05 | 不虚构真实企业客户、专家签核或真实学员效果 | M0 样例均为合成设计数据 | 明确标注“模拟企业/合成画像” |
| NG-06 | 不把 Docker 设为唯一运行路径 | 评委 Windows 环境可能无 Docker | 默认非 Docker 闭环；Docker 仅增强隔离执行 |
| NG-07 | 不在比赛阶段建设完整 HR/LMS/SSO 产品 | 会稀释核心垂直培训闭环 | 定义标准契约和模拟端点，后续再接入 |
| NG-08 | 不一次性导入全部 1.51 GiB GitHub 素材 | 许可、主题和质量未逐文件审查 | 白名单精选、保留来源与哈希 |
| NG-09 | 不把通用聊天回答当正式课程产物 | 缺少版本、审核和发布状态 | 生成 `artifact` 后通过质量门禁发布 |
| NG-10 | 不用前端静态拼接伪造 Agent 协同 | 无运行对象、不可回放 | 由后端事件和 `trace_id` 驱动界面 |
| NG-11 | 不因演示顺畅删除失败案例 | 破坏指标分母和真实性 | 保留失败，分类并给出恢复分支 |
| NG-12 | 不自动豁免高风险能力 | 历史成绩不能覆盖安全责任 | 强制人工审批或禁止免修 |

## 2. 证据等级

| 等级 | 名称 | 可支持的结论 | 最低元数据 | 不能单独支持 |
|---|---|---|---|---|
| E5 | 可复现运行证据 | 指定环境、代码、数据下功能/指标实际表现 | 提交哈希、数据版本、命令、原始输出、时间、环境 | 跨环境普遍成立、真实用户效果 |
| E4 | 自动测试证据 | 被测试契约在测试范围内通过 | 测试 ID、范围、结果、运行时间 | 未覆盖的端到端行为 |
| E3 | 连续人工操作证据 | 页面流程、状态变化和可观察结果 | 操作步骤、输入、截图/视频、预期与实际 | 内部算法正确、全量边界安全 |
| E2 | 源码直接证据 | 结构、分支、Schema 或算法实现存在 | 文件、行号/符号、版本 | 运行成功、效果达标 |
| E1 | 设计/文档证据 | 计划、契约、口径或人工声明存在 | 文档版本、责任人、评审状态 | 已实现、已测试、已达标 |
| E0 | 无法验证/冲突证据 | 只能记录待验证或矛盾 | 声明来源和缺口 | 任何肯定性技术结论 |

证据使用规则：结论强度不得高于证据。截图可以证明“界面显示过”，不能证明权限无漏洞；测试可以证明覆盖范围内通过，不能写成“所有功能绝对安全”；源码存在不能写成“评委环境已运行”。

## 3. 声明模板

每项对外声明必须记录：

```text
声明 ID：CLAIM-xxx
声明原文：
声明类型：结构 / 功能 / 性能 / 效果 / 合规 / 用户体验
适用范围与排除项：
最低证据等级：
证据位置：
代码/数据/模型版本：
当前状态：已证实 / 部分证实 / 未验证 / 被证伪
责任人和复核日期：
```

## 4. 功能完成定义（DoD）

功能只有同时满足以下八个维度，才能标记“完成”；某维度明确不适用时必须在 ADR 中说明。

| 维度 | 必须回答的问题 | 最低证据 |
|---|---|---|
| 数据 | 对象、所有者、版本、约束、迁移和删除如何处理？ | Schema/迁移测试 |
| API | 输入输出、错误、幂等、分页、追踪和兼容如何定义？ | OpenAPI/契约测试 |
| UI | 正常、加载、空、失败、无权、重试和恢复状态是否完整？ | E2E 或连续人工证据 |
| 权限 | 谁能对哪个对象做什么，跨用户/组织如何拒绝？ | 正向与负向自动测试 |
| 日志 | 能否按 `trace_id` 还原关键操作且不记录秘密？ | 结构化日志样例 |
| 测试 | 单元、集成、API、E2E 和人工案例覆盖了哪些风险？ | 新鲜测试报告 |
| 文档 | 用户、部署、接口、限制和失败恢复是否更新？ | 版本化文档 |
| 降级 | LLM、网络、索引、Docker 或外部服务不可用时会怎样？ | 故障注入或明确拒绝测试 |

## 5. DoD 检查表

```text
[ ] 需求有唯一 ID，并映射至 CAP-xx
[ ] 正常路径和至少一个失败分支有验收案例
[ ] 数据对象含 owner、版本、时间和状态约束
[ ] API 使用统一错误信封、trace_id 和时间格式
[ ] UI 不以静态假数据伪装后端成功
[ ] 学生、教师、管理员、越权和匿名边界均有判定
[ ] 日志不含 API key、密码、令牌和敏感正文
[ ] 自动测试实际运行，报告区分通过/失败/跳过/未运行
[ ] 手工验收记录输入、步骤、预期、实际和证据
[ ] 外部依赖不可用时可降级、重试或明确拒绝
[ ] 用户文档、API 文档、部署说明和追踪矩阵同步
[ ] 对外声明绑定不低于要求等级的证据
```

## 6. 阶段完成审计

阶段审计必须按“要求 -> 实现 -> 测试 -> 运行证据 -> 文档”顺序核对。发现冲突时以当前代码和新鲜运行结果为准，旧文档降级为历史证据。审计报告至少列出 `PASS/PARTIAL/FAIL`、证据路径、缺口、责任阶段和复核日期；“没有报错”“页面能打开”“AI 说完成了”都不是完成标准。
