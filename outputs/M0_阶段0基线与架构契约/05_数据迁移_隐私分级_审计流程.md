# 数据迁移、隐私分级与审计流程

版本：`M0-v0.1-draft`

## 1. 数据分级

| 等级 | 名称 | 例子 | 默认存储/访问 | 模型使用规则 |
|---|---|---|---|---|
| P0 | 公开课程素材 | 已核实许可的公开文档、课程目录、公开论文元数据 | 可随作品包分发；公开读取 | 可进入本地/授权模型上下文 |
| P1 | 项目与运行数据 | 课程版本、题目、系统指标、Agent 运行摘要 | 项目库；按组织/课程隔离 | 只传必要字段 |
| P2 | 个人学习数据 | 学号映射、作答、画像、错题、行为和路径 | owner/课程范围；默认不公开 | 脱敏、最小化、可关闭外部模型 |
| P3 | 敏感身份与企业数据 | 联系方式、真实人事信息、岗位考核和高风险安全记录 | 加密/严格审批；不进入比赛样例 | 默认禁止出境和训练；需明确授权 |
| P4 | 秘密 | API key、JWT secret、数据库凭据 | 环境变量/密钥管理器；绝不进 Git | 禁止进入日志、提示词和产物 |

## 2. 比赛样例数据规则

1. `datasets/` 中的学生和企业对象全部是合成数据，ID 使用 `student-m0-*`、`org-demo-*` 等不可回溯标识。
2. 不使用真实姓名、手机号、邮箱、学校学号或企业客户信息。
3. 课程素材只进入许可证已核实的白名单；没有许可证文件的仓库内容默认 `REVIEW_REQUIRED`，不能成为正式预置知识库。
4. 评测原始输出可以保存模型答案，但必须去除密钥、会话令牌和不必要的个人上下文。
5. 外部模型调用默认只传检索到的必要片段和脱敏画像；使用外部模型必须在运行报告中记录 provider/model/config，而不记录 API key。

## 3. 数据生命周期

| 数据 | 创建 | 保留 | 归档/删除 |
|---|---|---|---|
| 课程来源 | 引入时登记许可、哈希和来源 | 课程版本生命周期+历史证据期 | 许可失效时停止发布；正文可归档或删除 |
| 画像快照 | 诊断/决策时生成 | 与学习证据关联期间 | 学生请求删除；统计只留不可识别聚合 |
| Agent 运行 | 任务开始/完成时 | 评测和审计期 | 摘要保留，敏感输入按策略删除 |
| 原始作答 | 提交时 | 课程和申诉期 | 删除请求按法规执行；指标快照不可回写 |
| 审计日志 | 受控操作发生时 | 合规要求期 | 只允许合规归档/清除，不随普通对象删除 |
| 缓存/向量 | 摄入或生成时 | TTL 或索引版本期 | 与来源版本联动失效并清理 |

## 4. 向前迁移流程

```text
发现差异 -> 备份数据库/文件/索引 -> 生成迁移编号
-> 在空库和副本库演练 -> 校验约束与行数
-> 进入兼容期 -> 执行向前迁移/回填
-> 应用新旧兼容代码 -> 运行健康与回归测试
-> 发布 -> 观察 -> 淘汰旧字段
```

每个迁移必须包含：版本号、前置版本、脚本、回填规则、校验 SQL/脚本、回滚或恢复快照、预计停机、兼容窗口、负责人和演练结果。回滚优先恢复一致快照，不直接反向猜测业务数据。

## 5. 备份与恢复验收

- SQLite 主库、上传文件、向量索引、课程 manifest 和产物目录必须作为一个一致性备份集合。
- 备份文件不得包含 `.env` 或临时密钥。
- 恢复演练需验证：课程版本可读取、文档哈希不变、画像/作答关联仍完整、索引可重建、trace 查询可用。
- 只备份数据库而遗漏上传文件或索引，不算可恢复备份。

## 6. 隐私操作与审计

| 操作 | 前置检查 | 审计字段 | 结果 |
|---|---|---|---|
| 查看个人画像 | 身份、课程范围、最小字段 | actor、subject、purpose、trace | 允许/拒绝 |
| 导出学习报告 | 本人或授权角色、脱敏规则 | export_id、字段清单、接收范围 | 文件哈希与过期时间 |
| 删除个人数据 | 身份验证、关联对象盘点 | request_id、范围、完成时间 | 删除/匿名化清单 |
| 发布课程 | 来源许可、质量检查、人工责任 | version、review、audience | 可用版本 |
| 免修审批 | 风险等级、证据等级、审批人 | policy_version、evidence_refs | 通过/拒绝/过期 |
| 越权访问 | 统一拒绝处理 | actor、object_type、reason、trace | 不泄露对象存在性 |

日志只记录必要摘要；禁止记录密码、令牌、API key、完整敏感文档和未脱敏提示词。

## 7. M0 完成审计流程

1. **范围核对**：从实施计划和赛题提取要求，确认无遗漏。
2. **对象核对**：术语、API、事件、状态、数据库字段一一对应。
3. **证据核对**：为每一项要求绑定源码、测试、运行、人工或设计证据。
4. **数据核对**：课程来源、许可证、哈希、画像、企业样例和评测案例可追溯。
5. **真实性核对**：标出合成数据、未运行测试、未达标指标和未验证 Docker/外部模型。
6. **门禁评审**：产品、开发、测试和领域角色分别复核，不以单人自评作为全部证据。
7. **记录与整改**：生成 `17_M0阶段门禁审计报告.md`，缺口进入下一阶段，不在报告中隐去。

## 8. 当前明确的隐私风险

工作树存在 `.env` 和本地 SQLite/缓存文件。M0 文档只记录其存在与分级，不复制内容；提交/作品包必须通过密钥扫描和敏感文件清单，确保 `.env` 不进入压缩包。当前已有运行矩阵只能证明选定路由，不能替代全 API 合规审计。
