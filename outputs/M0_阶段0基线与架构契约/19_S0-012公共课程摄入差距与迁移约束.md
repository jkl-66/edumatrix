## 2. 根因

`DBKnowledgeDocument.student_id` 同时承担“所有者”和“可见范围”两个不同概念，并通过外键指向学生画像。学生上传文档可以使用该模型，平台公共课程却不是学生私有对象。现有代码虽然在列表查询中把 `public/system` 当作特殊所有者，但数据库模型没有对应的一等公共课程对象，形成以下冲突：

1. 公共课程需要被所有授权学生读取，但不应伪装成某个学生的私有文档。
2. 直接插入新的 `student_id='public'` 记录可能违反当前外键约束。
3. 创建一个假的“public 学生画像”只能绕过约束，不能建立课程、版本、章节、来源和发布状态。
4. 只修改当前 `edumatrix.db` 不能保证干净环境、源码包或评委机器可复现。

## 3. 禁止的临时做法

- 不关闭 SQLite 外键后强行插入公共记录。
- 不把 9 个文件绑定到某个真实或临时参赛账号。
- 不只修改提交时的本地数据库而缺少幂等初始化脚本。
- 不删除旧 25 章记录来伪造“替换完成”；必须有可回滚的迁移和版本边界。
- 不把“知识库页面可打开”写成“RAG 课程已摄入、可检索、可问答”。

## 4. 推荐迁移路径

M1 首个数据迁移应将课程定义与文档所有权分离，最低字段如下：

| 对象/字段 | 最低要求 |
|---|---|
| `course` | `course_id`、标题、版本、领域包、状态、可见范围 |
| `course_document` | `document_id`、`course_id`、章节顺序、来源、许可、哈希、解析状态 |
| 文档所有权 | `owner_type=user/course/system` 与 `owner_id`，不得继续用学生 ID 表示公共性 |
| 发布 | 草稿、已审核、已发布、已归档；学生只能看到已发布版本 |
| 摄入运行 | 幂等键为 `course_id + manifest_version + content_hash`，重复运行不得产生副本 |
| RAG/图谱 | 每个分块保留 `course_id`、`document_id`、章节、来源和许可元数据 |

迁移顺序：建立新表和兼容读取层 -> 导入 9 个 RAG 文件 -> 校验解析/分块 -> 发布课程版本 -> 前端按课程分组展示 -> 验证检索、引用、问答和图谱 -> 再决定旧 25 章记录的归档策略。
