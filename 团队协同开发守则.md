# EduMatrix 智教矩阵：8人团队协同开发与防混乱守则 (全量版)

> [!WARNING]
> **团队协作第一天条**：
> 在 8 人同时进行高强度开发时，最大的敌人不是“技术难度”，而是“代码冲突”、“数据库损坏”以及“AI 瞎改代码引发的全局雪崩”。
> 本守则为**强制执行规范**，全体成员在写下第一行代码前必须熟读并无条件死守。

## 🎯 核心指导：团队高并发协同“四大铁律”

为保障 8 人并行开发互不干扰，全员必须贯彻以下四大核心技术手段：
1. **契约先行**：为了彻底规避接口声明文件的合并冲突，API 已经物理拆分。**每当组员要开发新功能时，必须先去 [frontend/src/api/](frontend/src/api/) 目录下自己模块对应的 API 文件（如 [quiz.js](frontend/src/api/quiz.js)、[profile.js](frontend/src/api/profile.js)）中声明新接口并写好 Mock 格式，然后再写后端算法。** 这能实现接口层面的完美物理隔离。
2. **物理隔离**：后端路由独立，前端页面独立。**允许子功能开发者自行在公共页面（如 `Chat.vue`）中注册并挂载自己的组件标签与 import（仅限 3 行注册代码）**。但其 100% 的业务与样式代码必须写在自己的独立组件文件中，且 CSS 强制 scoped，以实现 Git 智能无冲突自动合并。
3. **绿色提交**：本地测试 `pytest test_edumatrix.py` 全绿通过才允许推送。PR 开启同行 Peer 互审合并，组长离线时对接人直接合并，绝不排队。
4. **统一基底**：教材文件与数据库预置数据完全统一。新环境冷启动按照 `下载教材` -> `编译向量库` -> `预置数据` 的三步标准指令运行，严禁私自增删本地课件。

## 🛑 第一章：Git 分支管理与代码冲突防御 (Git & Merge Conflict)

### 🚨 常见灾难场景
*   *场景*：成员 A 和成员 B 同时修改了 `app/main.py` 或 `models.py`，合并分支时产生了几百行的代码冲突，误删了别人的核心算法。

### 🛠️ 团队硬性规则
1. **禁止直接推送 `main` 分支**：`main` 分支是神圣不可侵犯的生产线，任何直接的 `git push origin main` 均视为违规。
2. **严格的分支命名规范**：
   *   所有开发必须在独立分支进行。命名格式为：`feat/姓名缩写-模块名`（如：`feat/lzz-profile` 或 `feat/wy- RAG`）。
3. **每日清晨 Merge/Rebase**：
   *   每天早上开工前，第一件事必须是：`git checkout main` -> `git pull` -> `git checkout 自己的分支` -> `git merge main`（或 `git rebase main`）。
   *   保证自己的本地开发分支与线上主分支的差异不超过 24 小时。
4. **Pull Request (PR) 自助直接合并制（无冲突直通车）**：
   *   功能写完并测试全绿后，发起 PR 合并到 `main`。
   *   **无冲突时**：如果 GitHub 显示没有代码冲突（No conflicts），**开发者可直接自助点击 Merge 合并入主干**，无需等待组长或对接人审查，最大化开发效率。
   *   **有冲突时**：如果 GitHub 提示冲突，合并自动挂起。开发者必须使用 `git log` 查看是谁修改了冲突行，私聊该成员并在本地共同解决冲突后，重新推送，再行合并。

### 📅 成员每日 Git 协同工作流 (Daily Developer Workflow)

全员开发时，请严格按照以下“上午、中午、傍晚、提交”的节奏进行协同，以实现零冲突合并：

#### ☀️ 上午：同步与对齐（防冲突第一步）
*   **第一步**：切换回主分支，拉取最新代码：
    ```bash
    git checkout main
    git pull origin main
    ```
*   **第二步**：切换回自己的分支，把主分支最新的改动合并进来：
    ```bash
    git checkout feat/lmm-profile  # 替换为自己的分支名
    git merge main
    ```
    *(注：如果此时发生冲突，在本地立刻解决，不要拖到提交 PR 时)*

#### 💻 中午：本地安心开发
*   **第三步**：在自己的分支上开发，AI 辅助编写代码。
*   **第四步**：本地完成阶段性功能后，本地提交（可以多次提交）：
    ```bash
    git add .
    git commit -m "feat: 实现了时序知识追踪模型数据解析"
    ```

#### 🌙 傍晚：测试与推送
*   **第五步**：**测试过关（最重要）**。在根目录下运行全量测试，检查 AI 是否破坏了已有功能：
    ```bash
    pytest test_edumatrix.py
    ```
    *(必须全部显示绿色 PASS 才能进行下一步！)*
*   **第六步**：将自己的开发分支推送到 GitHub 远程仓库：
    ```bash
    git push origin feat/lmm-profile  # 替换为自己的分支名
    ```

#### 🚀 发起合并申请（PR）
*   **第七步**：登录 GitHub，发起从自己的开发分支合并到 `main` 的 Pull Request (PR)。
*   **第八步**：**自助合并与冲突协同**。若没有冲突且本地 `pytest` 全绿，**直接在 GitHub 上点击 Merge 合并入主干**。如果产生冲突，根据 `git log` 私聊联系冲突处的对应成员，在本地合并解决冲突后再次推送。

## 🗄️ 第二章：数据库结构（Schema）变更锁定协议

### 🚨 常见灾难场景
*   *场景*：成员 1 修改了 `student_profiles` 表，添加了字段，但没有通知大家。成员 6 拉取代码运行，SQLite 报错 `no such column` 导致系统崩溃，排查了半天无果。

### 🛠️ 团队硬性规则
1. **“Schema 变更锁”机制**：
   *   任何人修改 `models.py` 里的 SQLAlchemy 实体类（即修改数据库字段、新增表）之前，必须在微信群里发出公告，得到组长确认。
2. **提供增量或重建脚本**：
   *   由于我们使用 SQLite WAL 模式，开发环境下结构变更时，修改人必须提供**清库重建数据库的简单命令**或**迁移脚本**（如 `python scripts/reinit_db.py`）。
3. **隔离开发数据**：
   *   禁止将本地带有个人测试数据的 `edumatrix.db` 提交到 Git（必须在 `.gitignore` 中屏蔽 `edumatrix.db`, `*.db-wal`, `*.db-shm`）。每个人本地的数据库文件都是独立生成的。

## 🧪 第三章：AI 辅助开发（Vibecoding）安全防线

### 🚨 常见灾难场景
*   *场景*：成员 5 完全依靠 AI 写完了一个类，自己没看懂，随手 merge 到了 main 分支。结果由于该类在特定边界情况下会返回 None，导致其他人的页面全部白屏。

### 🛠️ 团队硬性规则
1. **“绿灯测试（Green Build）”推行（已配置 Git Hook 自动强锁）**：
   *   根目录下有全量测试用例 `test_edumatrix.py`。
   *   **Git 提交强锁定**：**项目已内置 Git Pre-Commit Hook 预检脚本。当组员执行 `git commit` 时，系统会自动激活虚拟环境并在后台跑一遍测试。** 如果测试失败，提交将被强行拦截。请修复测试后再行提交。你也可以手动运行 `pytest test_edumatrix.py` 调试报错。
2. **AI 代码审计义务**：
   *   谁用 AI 生成代码，谁就必须对这行代码的**每一行逻辑负有终身解释权**。
   *   禁止把“不知道为什么，反正 AI 给的代码能跑通”作为提交借口。必须至少看懂代码的数据结构和异常处理。

## 🎨 第四章：前端样式（CSS）与全局组件冲突规避

### 🚨 常见灾难场景
*   *场景*：成员 7 修改了 [style.css](frontend/src/style.css) 里的全局样式，导致成员 8 辛辛苦苦调好的数据大盘排版全乱，按钮错位。

### 🛠️ 团队硬性规则
1. **必须使用 Scoped CSS**：
   *   除了全局的公共 Token（由成员 7 维护在 [style.css](frontend/src/style.css) 中），其他所有 `.vue` 组件的 `<style>` 标签，**必须强制加上 `scoped` 属性**：
     ```vue
     <style scoped>
     /* 你的局部样式 */
     </style>
     ```
2. **严禁修改他人页面的 Class 命名**：
   *   在未取得对方同意前，不要为了“顺眼”去重隔别人的组件 DOM 结构或全局 Class 名。
3. **UI 规范一致性**：
   *   卡片的阴影、渐变底色、圆角大小统一调用成员 7 在 [style.css](frontend/src/style.css) 中定义的全局 CSS 变量（如 `--color-primary`, `--blur-glass`），禁止自己乱写 ad-hoc 的色值。

## 🔑 第五章：API 密钥泄露与大模型流控安全

### 🚨 常见灾难场景
*   *场景*：成员 3 把写有自己私有 API Key 的 `.env` 文件提交到了 GitHub 公开仓库，导致 Key 瞬间被黑客扫走，欠费几千元。

### 🛠️ 团队硬性规则
1. **绝对隔离 `.env` 文件**：
   *   本地创建的 `.env` 配置文件绝对不允许进入 Git。
   *   拉取新代码后，只能复制 `.env.example`，命名为 `.env`，然后在本地填入自己的 API Key。
2. **Prompt 异常拦截 (Graceful Degradation)**：
   *   调用大模型的方法必须包裹在 `try...except` 块中。
   *   当 LLM API 由于欠费、断网或被限流返回错误时，API 必须返回友好的 Mock 兜底数据（如返回缓存笔记或假题目），**坚决不允许直接抛出 500 崩溃给前端**。

   *   **严禁擅自修改大模型 JSON 返回字段的名称和类型**。
   *   如果确需修改，必须先修改 `models.py` 里的 Pydantic 约束类或 Dataclass 类，并通知受此数据流影响的上下游成员同步修改解析代码。

## 🚨 第七章：团队应急响应流程 (Emergency Protocol)

当你在本地遇到“代码跑不起来”、“拉下最新代码后报错”时，严格按照以下步骤操作，**严禁在群里盲目发泄情绪或盲目改动他人代码**：

1. **第一步：排查本地环境**
   *   运行 `git status` 检查是否有未提交的脏代码。
   *   检查本地的 `.env` 配置是否与最新的 `.env.example` 一致。
2. **第二步：本地重置测试**
   *   如果怀疑是数据库结构坏了，删除本地的 `edumatrix.db`，运行数据库重建脚本，重新拉起服务测试。
3. **第三步：精确定位嫌疑人**
   *   使用 `git log` 查看最近 3 次是谁提交了合并。
   *   私聊对应的修改人，协同解决，切勿擅自在本地把别人的代码注释掉。

## 🚀 第八章：本地环境冷启动与未入 Git 文件清单

以下文件已被 `.gitignore` 排除，克隆项目后**不会自动出现在本地**。每个成员必须手动或运行脚本在本地生成，否则系统将因“找不到文件/配置”而报错。

### 1. 🔑 环境变量配置文件 (`.env`)
*   **为何不在 Git 中**：包含大模型的 API 私钥和本地配置，属于机密。
*   **冷启动方法**：
    1. 在项目根目录下，复制一份 `.env.example` 并重命名为 `.env`。
    2. 打开 `.env`，填入分发的大模型 API 密钥（如 `LLM_API_KEY`）及服务商地址，保存即可。

### 2. 🗄️ 本地 SQLite 数据库文件 (`edumatrix.db`)
*   **为何不在 Git 中**：包含本地测试生成的临时学生数据、对话历史。
*   **冷启动方法**：
    1. 启动 FastAPI 后端服务时（运行 `python run.py`），系统会在 `startup` 事件中自动执行 `init_db()` 创建此文件。
    2. 为了生成初始测试学生和教师数据，必须在终端运行一次数据预置脚本：
       ```bash
       python scripts/seed_students.py
       ```

### 3. 🧠 FAISS 向量检索索引目录 (`data/` 文件夹)
*   **为何不在 Git 中**：向量索引属于大型二进制文件，且随着上传资料会增量热更新，GitHub 无法高效托管。
*   **冷启动方法**：
    1. 克隆代码后，你的 `data/` 目录将是空的，**直接发起 RAG 知识检索会报错或返回空值**。
    2. 必须运行一次本地教材下载脚本，拉取预设的标准教参库（共 13 个 GitHub 教材仓库）：
       ```bash
       python scripts/collect_ml_multimodal_data.py --download
       ```
    3. 运行本地知识向量化管道，提取刚才下载的教材并编译本地索引：
       ```bash
       python scripts/ingest_to_faiss.py
       ```
       等待脚本提示“FAISS index created successfully at data/”后，知识库方可正常使用。

### 4. 📦 前后端第三方依赖库 (`node_modules` & `venv`)
*   **为何不在 Git 中**：第三方包体积极大，统一由包管理器拉取。
*   **冷启动方法**：
    *   **后端 Python 依赖**：激活虚拟环境后，在根目录运行 `pip install -r requirements.txt`。
    *   **前端 Vue 依赖**：在终端进入 `frontend` 目录，运行 `npm install` 安装所有前端库。

## 📝 第九章：个人开发日志 (WORKLOG.md) 维护规范

为了清晰追踪 8 位成员的每日研发进展，避免信息不对称，全体成员必须贯彻开发日志的维护规范：

### 🛠️ 团队硬性规则
1. **“每日记录，落盘为安”**：
   *   项目根目录下已创建统一的 [WORKLOG.md](WORKLOG.md) 文件。
   *   每位成员**在每天下班前，或完成某项阶段性功能后**，必须在该文件中追加自己的开发日志。
2. **日志书写格式**：
   *   必须包含：**日期**、**姓名/模块**、**今日已完成 (Completed)**、**明日计划 (Planned)**。
   *   如果遇到卡点或需要他人配合的 Block 风险，也需显式注明。
   *   *示例*：
     ```markdown
     ### 📅 2026-07-02
     * **wy (RAG 检索模块)**:
       * **已完成**: 实现了 `webSearch.js` 拆分，完成了 ColPali 图表多模态检索缓存优化。
       * **计划**: 明天对接前端 RAG 历史列表渲染。
       * **Block风险**: 无。
     ```
3. **Commit 提交关联**：
   *   在每次发起合并 PR 之前，必须在自己的分支中将 [WORKLOG.md](WORKLOG.md) 写入并提交。
   *   **严禁在没有更新 `WORKLOG.md` 的情况下强行发起 PR。**
